# Phase 3 Word Export Extraction Plan

Date: 2026-05-29

Status: Draft

Source of truth for extraction: the UCAS Word export workflow in the main `ChineseResearchLaTeX` project, especially `projects/thesis-ucas-doctor/scripts/export_docx.py` and its supporting format-quality tools.

## Purpose

Phase 3 should productize a small, stable subset of the main UCAS project's Word export workflow into this Delivery Kit. It should not copy the full UCAS thesis postprocessing chain.

The Kit currently provides:

- a beta `pandoc` DOCX wrapper through `python scripts/ucas.py export-docx`
- TeX preprocessing through `prepare-tex`
- format quality checks and repairs through `check-format-quality` and `fix-format`
- a full-mode smoke path that can call a thesis project's own `scripts/export_docx.py` when present

The gap is not "no Word workflow". The gap is that advanced Word behavior is still project-local and not available as small, tested Kit commands.

## Extraction Principles

- Extract narrow utilities with explicit contracts.
- Keep UCAS-specific layout behavior in the main project unless it is parameterized.
- Prefer fixtures and tests before moving implementation.
- Keep optional Windows/Office behavior behind environment-gated flags.
- Make portable behavior useful without Microsoft Word.
- Preserve the Kit's role as a delivery toolkit, not a full thesis template.

## Candidate PR Sequence

### PR 1: Phase 3 Planning and Fixtures

Goal: make the extraction target testable before implementation.

Files likely to change:

- `docs/development/v0.3.0-phase3-extraction-plan.md`
- `docs/word-export/README.md`
- `examples/thesis-project/`
- `tests/`

Scope:

- Add synthetic fixtures for mixed Chinese/English bibliography, one bilingual figure, one bilingual table, and one minimal DOCX field-update fixture if it can be generated without private content.
- Document supported and unsupported Phase 3 behavior.
- Add placeholder tests or smoke tests that describe the expected public API.

Acceptance:

```bash
python -m pytest tests/ -v
python scripts/ucas.py check-format-quality --project-dir examples/thesis-project --mode fast
python scripts/ucas.py fix-format --project-dir examples/thesis-project --dry-run
```

### PR 2: Optional Word Field Update Helper

Goal: expose a safe optional helper for updating DOCX fields with Microsoft Word.

Candidate command:

```bash
python scripts/ucas.py update-word-fields --docx path/to/main.docx
```

Candidate flags:

- `--visible`
- `--timeout <seconds>`
- `--dry-run`

Reusable source concepts from the main project:

- Windows platform preflight.
- `powershell`/`pwsh` discovery.
- Word COM automation through a temporary output file.
- DOCX package integrity validation before replacing the original file.

Do not extract:

- UCAS header/footer synchronization.
- UCAS catalog fallback logic.
- Word OpenAndRepair as a default behavior until it has a small, isolated contract.

Acceptance:

- Non-Windows test verifies early, clear failure when not supported.
- Dry-run test reports environment capability without mutating the DOCX.
- Windows manual smoke verifies TOC/LOF/LOT fields update and the original is not replaced if validation fails.

### PR 3: CSL and BibTeX Support

Goal: make Pandoc citation export more reliable without importing the full UCAS bibliography postprocessor.

Candidate changes:

- Add a Kit-owned GB/T 7714 author-date CSL path or documented external CSL lookup.
- Extend `scripts/word_export/pandoc_export.py` to support `--citeproc`, `--csl`, and repeated `--bibliography`.
- Add BibTeX sanitization for fields known to degrade Pandoc DOCX output.

Candidate command:

```bash
python scripts/ucas.py export-docx \
  --project-dir examples/thesis-project \
  --bibliography examples/thesis-project/bibs/references.bib \
  --csl docs/word-export/china-national-standard-gb-t-7714-2015-author-date.csl
```

Do not extract:

- UCAS-specific bibliography paragraph layout.
- Chinese/English author-term text rewrites that depend on final DOCX structure.
- Citation audit rules that require `.bcf`, `.bbl`, and PDF text unless they are separately documented as quality checks.

Acceptance:

- Pandoc command construction test includes `--citeproc`, `--csl`, and sanitized temporary bibliography paths.
- Mixed Chinese/English `.bib` fixture remains valid after sanitization.
- Missing bibliography file fails with a clear message.

### PR 4: Caption Marker Minimal Contract

Goal: define a small caption marker protocol that can survive Pandoc and be postprocessed predictably.

Reusable source concepts from the main project:

- Caption marker shape.
- Caption text cleanup.
- Figure/table sequence formatting.
- Appendix-aware numbering only if covered by fixtures.

Do not extract:

- Full UCAS bilingual caption typography.
- Landscape figure special handling.
- Full figure/table catalog reconstruction.
- Arbitrary LaTeX macro support.

Acceptance:

- One figure fixture renders a stable marker.
- One table fixture renders a stable marker.
- Appendix fixture either passes with documented numbering or is explicitly out of scope.
- Unsupported macro fixture degrades gracefully with a diagnostic.

### PR 5: Lightweight DOCX Postprocess Shell

Goal: provide a Kit-level postprocess entry point that can run small generic DOCX checks and future helpers.

Candidate command:

```bash
python scripts/ucas.py postprocess-docx --docx path/to/main.docx --report dist/docx-report.md
```

Initial scope:

- Verify DOCX package integrity.
- Generate a small report.
- Optionally call caption marker replacement if PR 4 lands.
- Optionally call Word field update helper if PR 2 lands and the user requests it.

Do not extract:

- UCAS cover metadata synchronization.
- UCAS front matter and back matter section rules.
- UCAS header/footer and page-number synchronization.
- Full quality report parity with `projects/thesis-ucas-doctor/scripts/export_docx.py`.

Acceptance:

- Existing DOCX fixture produces a report.
- Invalid DOCX fails before mutation.
- Running twice is idempotent for supported operations.

## Deferred Main-Project-Only Capabilities

These should remain in `ChineseResearchLaTeX` until a profile system exists:

- UCAS cover metadata synchronization.
- UCAS front/back matter section detection.
- UCAS page-number restart and odd-page section repair.
- UCAS header/footer rules.
- UCAS typography overrides.
- Full figure/table catalog page-number materialization.
- Word OpenAndRepair as part of a large postprocessing chain.
- Static replacement of complex thesis cross references beyond the documented caption marker contract.

## Skill-Assisted Review Gate

The local `auto-test-code` skill is applicable to Phase 3 code PRs in this Kit.
It should not be used as a substitute for the fixture-first implementation plan,
but it is useful as a structured self-review gate once a PR changes Python source,
tests, CLI behavior, DOCX handling, or packaging logic.

Use it for:

- PR 2: optional Word field update helper.
- PR 3: CSL/BibTeX support.
- PR 4: caption marker minimal contract.
- PR 5: lightweight DOCX postprocess shell.
- Any PR that changes `scripts/`, `tests/`, `.github/`, packaging, or security-sensitive file handling.

Do not require it for:

- Documentation-only planning PRs.
- README link-only changes.
- Release-note edits that do not change behavior.

Expected workflow:

1. Create the skill workspace under the Kit repository root:

   ```bash
   python <auto-test-code-skill>/scripts/create_session.py --code-root . --run-id run_YYYYMMDDHHMMSS --kind a --id vYYYYMMDDHHMM
   ```

2. Perform the A-round critical code review against the current PR state. Keep all review artifacts under `tmp/run_YYYYMMDDHHMMSS/tests/vYYYYMMDDHHMM/`.
3. Fix accepted P0/P1 findings in source files and record evidence in the A-round `TEST_RUN.md` and `TEST_REPORT.md`.
4. Create the B-round quality session:

   ```bash
   python <auto-test-code-skill>/scripts/create_session.py --code-root . --run-id run_YYYYMMDDHHMMSS --kind b --id vYYYYMMDDHHMM --a-test-id vYYYYMMDDHHMM
   ```

5. Run the standard Kit verification commands and record the results in the B-round report.
6. Keep `tmp/` out of release packages and commits unless a future maintenance task explicitly asks to preserve review artifacts.

Minimum verification commands for code PRs:

```bash
python -m pytest tests/ -v
python scripts/ucas.py --help
python scripts/ucas.py check-format-quality --project-dir examples/thesis-project --mode fast
python scripts/ucas.py fix-format --project-dir examples/thesis-project --dry-run
python scripts/ucas.py check-privacy --project-dir .
python scripts/ucas.py pack --project-dir . --dry-run
```

## Release Planning and Skill Use

Use the following local skills and project gates when preparing, reviewing, and
publishing Phase 3 work. Skills are not project dependencies; they are execution
aids for the maintainer/agent session.

| Stage | Applicable skill or gate | Use in this project | Notes |
| --- | --- | --- | --- |
| Multi-step release planning | `plan-progress-logging` | Optional for long release trains or resumed work | Use the existing development plan for small PRs; create a separate progress log only when decisions span multiple sessions. |
| Code PR self-review | `auto-test-code` | Required for PRs that change Python code, CLI behavior, DOCX handling, packaging, or security-sensitive logic | Keep artifacts under `tmp/run_*/tests/`; do not commit them. |
| Whole-project release hardening | `auto-test-project` | Optional before a large v0.3.0 release candidate | Use only when the user explicitly asks for project-level testing; it is heavier than the standard release gate. |
| Secrets and privacy handling | `secrets-hygiene` plus `check-privacy` | Required whenever release logs, GitHub CLI auth, config files, or package contents may expose tokens or private paths | Never print tokens; rely on `python scripts/ucas.py check-privacy --project-dir .` before packaging. |
| Runtime failure during release | `experience-first-troubleshooting` | Required when repeated test, pack, GitHub CLI, Pandoc, LaTeX, or Word automation failures appear | Search prior troubleshooting notes before trying unrelated fixes. |
| Pre-completion evidence | `verification-before-completion` | Required before claiming implementation, tests, packaging, PR, or release are complete | Run fresh commands and report actual output, not assumptions. |
| Code review before merge | `requesting-code-review` or a bounded subagent review | Required for nontrivial code PRs before merge | Fix Critical/Important findings or document why they do not apply. |
| Branch completion | `finishing-a-development-branch` | Use after implementation is complete and tests pass | Present merge/PR/keep/discard options after verification. |
| Commit preparation | `git-commit` | Use only when the user asks to commit | It may auto-stage/push by default; verify staged files first and keep `HANDOFF.md` out unless explicitly intended. |
| GitHub Release publishing | `git-publish-release` | Use only when the user asks to create a GitHub Release or release notes | Requires `gh auth status`, an existing tag, and release notes based on actual commits. |
| PR decision review | `git-pr-review` | Use only when asked to review an existing GitHub PR from the outside | Do not use it for local implementation or trusted local branches. |
| Multi-agent implementation | `awesome-code` / subagents | Use when tasks are independent enough to split | For Phase 3, good split points are fixtures, Word helper, CSL/BibTeX, caption markers, and postprocess shell. |

Recommended release implementation order:

1. Finish the planned Phase 3 PR slice.
2. Run `auto-test-code` for code-changing PRs, then the standard verification commands.
3. Run `check-privacy` and `pack --dry-run` before any package or release claim.
4. Request code review before merging nontrivial implementation.
5. Use `finishing-a-development-branch` to choose local merge, PR, or keep-branch workflow.
6. After merge and tag creation, use `git-publish-release` only if publishing to GitHub is explicitly requested.

Optional machine-dependent checks:

```bash
python scripts/ucas.py export-docx --project-dir examples/thesis-project --output dist/example.docx
python scripts/ucas.py check-format-quality --project-dir examples/thesis-project --mode full
```

The full-mode check may require the source thesis project layout and external tools. It should be documented as optional unless the required project-local `scripts/export_docx.py`, Pandoc, LaTeX, and Word dependencies are all present.
